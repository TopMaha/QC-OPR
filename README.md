# QC OPR — ระบบติดตามการผลิต QC แบบเรียลไทม์

ติดตาม **ยอดผลิต (OPR)** + **ผล QC (ผ่าน/ของเสีย)** ราย **Valve No** แยกกะ
แล้วดู **Dashboard เรียลไทม์** ราย **ลูกค้า** ว่ามีงานต้องส่ง (ขาย) วันไหน ขาดอีกเท่าไร ในจอเดียว

- 🔐 **Lock-in ด้วยรหัสพนักงาน** — เข้าระบบด้วย *รหัสพนักงาน* เท่านั้น · ชื่อผู้บันทึกเติมอัตโนมัติจากการเข้าระบบ
- 📱 **Single-file PWA** — ทั้งแอปอยู่ใน [`index.html`](index.html) ไฟล์เดียว เปิดได้ทันที
- 🔌 **Offline / เดโม** — ทำงานได้เลยโดยเก็บข้อมูลใน `localStorage` (ไม่ต้องมี backend)
- ☁️ **ออนไลน์** — ใส่ URL ของ Cloudflare Worker ในหน้าตั้งค่า → ซิงก์ทุกอุปกรณ์

## โมเดลข้อมูล & สถานะการส่ง

- **พนักงาน** `{รหัส, ชื่อ}` — ใช้เข้าระบบ + เป็นผู้บันทึกอัตโนมัติ
- **Valve** `{Valve No, ลูกค้า, สินค้า, route(process), stages}` — Process ตั้งราย valve ได้ (ไม่เหมือนกัน)
- **แผนงาน (1 แถว)** `{Valve No, Lot, จำนวน, วันที่ขาย}` — นำเข้าจาก Excel
- **บันทึก** `{วันที่, กะ, Valve No, Process, Lot, OPR, NG, ผ่าน, ผู้บันทึก}`

แต่ละ Valve คำนวณสด: `ผลิตแล้ว = Σ OPR ของ valve` · `ขาดขาย (เหลือส่ง) = จำนวนเป้า − ผลิตแล้ว` · `เหลือกี่วัน = วันที่ขาย − วันนี้`

| สถานะ | เงื่อนไข | สี |
|---|---|---|
| Done (เสร็จ) | ผลิตครบเป้า | เขียว |
| In progress (กำลังผลิต) | ผลิตบางส่วน ยังไม่ถึงกำหนด | น้ำเงิน |
| Pending (รอเริ่ม) | ยังไม่เริ่มผลิต | เทา |
| Due soon (ใกล้ถึงกำหนด) | เหลือ ≤ 3 วัน ยังไม่ครบ | ส้ม |
| Overdue (เลยกำหนด) | วันนี้ > วันที่ขาย ยังไม่ครบ | แดง |

สถานะลูกค้า = ความรุนแรงสูงสุดของ valve ในมือ · แถบเตือนบนแดชบอร์ดนับ Overdue + Due soon

## ต่อออนไลน์ — ทุกเครื่องเห็นข้อมูลชุดเดียวกัน (Cloudflare Worker + D1)

> ✅ **เปิดใช้งานจริงแล้ว:** https://qc-opr.wiphawas-sketchup.workers.dev
> ฐานข้อมูล D1 `qc-opr` (`80cb9b3c-feec-4ab4-a322-52dac91831a2`) · ตารางครบ 5 (employees/valves/plans/records/targets)
> **ครั้งแรกที่เปิด** ฐานข้อมูลยังว่าง → เข้าระบบด้วย **PIN ผู้ดูแล `qcopr1234`** แล้วไปหน้า *ตั้งค่า* เพิ่มพนักงาน/Valve, หน้า *แผนงาน* วางข้อมูลจาก Excel — ทุกอย่างบันทึกลง D1 ให้เอง

ทำครั้งเดียว แล้ว **เปิด URL ของ Worker บนเครื่องไหนก็เห็นข้อมูลชุดเดียวกัน** เพราะ Worker เสิร์ฟทั้ง
ตัวแอป (`index.html`) และ API จากที่เดียว → แอปต่อ D1 ให้อัตโนมัติแบบ *same-origin* (ไม่ต้องกรอก URL เอง)

```bash
# 0) ติดตั้ง + ล็อกอิน Cloudflare (ครั้งแรกครั้งเดียว)
npm install -g wrangler
wrangler login

# 1) สร้างฐานข้อมูล D1
wrangler d1 create qc-opr
#    คัดลอก database_id ที่ได้ → วางแทน "PUT-YOUR-D1-DATABASE-ID-HERE" ใน wrangler.toml

# 2) สร้างตารางใน D1 (ตาม schema.sql)
wrangler d1 execute qc-opr --remote --file=./schema.sql

# 3) deploy Worker (เสิร์ฟทั้งแอป + API + ต่อ D1)
wrangler deploy
#    จะได้ URL เช่น https://qc-opr.<subdomain>.workers.dev
```

จากนั้น **เปิด URL ของ Worker** บนทุกเครื่อง → แถบมุมขวาบนจะขึ้น **ออนไลน์** และดึงข้อมูลจาก D1 มาแสดงให้เอง
(เพิ่ม/แก้/ลบ จากเครื่องไหน อีก ~15 วินาทีเครื่องอื่นก็เห็น)

> - `wrangler.toml` เปิด `[assets]` ไว้แล้ว — Worker จึงเสิร์ฟ `index.html` ให้ ส่วน `.assetsignore` กันไม่ให้เสิร์ฟไฟล์ backend/config
> - อยากต่อจากหน้าเว็บที่โฮสต์แยก (เช่น GitHub Pages) ก็ยังได้: หน้า **ตั้งค่า** → ช่อง **API Base URL** ใส่ URL ของ Worker → กด *เชื่อมต่อ*
> - เปิดไฟล์ `index.html` ตรง ๆ (`file://`) หรือโฮสต์ static ที่ไม่มี backend = **โหมดเดโม** อัตโนมัติ (เก็บในเครื่อง)

### Endpoints
| Method | Path | ทำอะไร |
|---|---|---|
| GET | `/api/init` | คืน `{ employees, valves, plans, records, targets }` |
| POST | `/api/employees` | upsert 1 พนักงาน |
| DELETE | `/api/employees/:code` | ลบพนักงาน |
| POST | `/api/valves` | upsert 1 valve (route/stages เป็น JSON) |
| DELETE | `/api/valves/:id` | ลบ valve |
| POST | `/api/plans` | upsert 1 แถวแผนงาน (valveNo/lot/qty/sellDate) |
| DELETE | `/api/plans/:id` | ลบแถวแผนงาน |
| POST | `/api/records` | upsert 1 record (valveNo/process/...) |
| DELETE | `/api/records/:id` | ลบ record |
| POST | `/api/targets` | upsert KV ตัวเลข |
