# QC OPR — ระบบติดตามการผลิต QC แบบเรียลไทม์

ติดตาม **ยอดผลิต (OPR)** + **ผล QC (ผ่าน/ของเสีย)** ต่อ *เครื่อง–process–ไลน์* แยกกะ
แล้วดู **Dashboard เรียลไทม์** ของทั้งโรงงานในจอเดียว

- 📱 **Single-file PWA** — ทั้งแอปอยู่ใน [`index.html`](index.html) ไฟล์เดียว เปิดได้ทันที
- 🔌 **Offline / เดโม** — ทำงานได้เลยโดยเก็บข้อมูลใน `localStorage` (ไม่ต้องมี backend)
- ☁️ **ออนไลน์** — ใส่ URL ของ Cloudflare Worker ในหน้าตั้งค่า → ซิงก์ทุกอุปกรณ์
- 🔁 **ซิงก์ข้ามอุปกรณ์** — process ของแต่ละเครื่องซิงก์ผ่าน KV store ด้วยแพทเทิร์น numeric-encode

## หน้าจอ

| หน้า | ทำอะไร |
|---|---|
| 📝 **บันทึก** | เลือกเครื่อง (auto-fill ไลน์/process) → ใส่ OPR + ของเสีย → `ผ่าน` คำนวณให้ · แยกกะเช้า/ดึก · พิมพ์ Lot ที่มีแผน → โชว์ "เหลือเท่าไร/กำหนดส่ง" |
| 📊 **แดชบอร์ด** | **ติดตามรายตัว Valve No** — แต่ละ Valve แสดงการไหลผ่านกระบวนการ (stepper: กลึง › เจาะ › QC › FG) สีตามสถานะแต่ละขั้น · KPI เสร็จ/กำลังทำ/เลยกำหนด · แตะขั้นเพื่อเปลี่ยนสถานะ · refresh อัตโนมัติ · **แถบเตือน "งานด่วน"** |
| 📋 **แผนงาน** | วางแผนต่อ Lot (เป้าหมาย + กำหนดส่ง) · ความคืบหน้าดึงจากยอด OPR จริง · เตือนความด่วนอัตโนมัติ |
| 🕓 **ประวัติ** | ค้นหาตามวันที่/เครื่อง/Lot · แก้ไข/ลบรายการ |
| ⚙️ **ตั้งค่า** | เชื่อมต่อคลาวด์ · จัดการเครื่อง+process (ซิงก์) · เพิ่มชื่อ process · สำรอง/นำเข้า/ล้างข้อมูล |

**เกณฑ์สี** (ปรับใน `CONFIG` ของ `index.html`): ของเสีย ≤ 3% เขียว · ≤ 8% ส้ม · เกินนั้นแดง · ไม่มีข้อมูล เทา

### ติดตามตาม Valve No (แดชบอร์ด) & แก้ไขกระบวนการ
แต่ละ **Valve No** มี **route** = ลำดับกระบวนการที่ต้องผ่าน (เช่น กลึง → เจาะ → เชื่อม → QC → FG)
แต่ละขั้นมีสถานะ: `ยังไม่เริ่ม` → `กำลังทำ` → `เสร็จ` แสดงเป็น stepper สีตามสถานะ + ขั้นปัจจุบันมีกรอบไฮไลต์

- **แตะที่ขั้น** บนการ์ด → วนสถานะทันที (อัปเดตเร็วหน้างาน)
- **✏️ แก้ไขกระบวนการ** → เพิ่ม/ลบ/สลับลำดับกระบวนการ + ตั้งสถานะแต่ละขั้น + กำหนดส่ง (= "ระบบแก้ไขกระบวน Valve No")
- สถานะ Valve: เสร็จ (ครบทุกขั้น) · กำลังทำ · ยังไม่เริ่ม · เลยกำหนด (เลย dueDate และยังไม่ครบ) — เรียงเลยกำหนด/กำลังทำขึ้นบน

### การวางแผน & "รู้ว่างานจะด่วน" (urgency engine)
แต่ละแผน = 1 **Lot** + **เป้าหมาย (ชิ้น)** + **กำหนดส่ง** — ความคืบหน้าดึงจากยอด OPR ที่บันทึกของ Lot นั้นโดยอัตโนมัติ
(บันทึกผลผลิตด้วย Lot เดียวกัน → แผนขยับเอง) ระบบคำนวณสด:

- `เหลืออีก = เป้า − ผลิตแล้ว` · `เหลือกี่วัน = กำหนดส่ง − วันนี้`
- `ต้องเร่งวันละ = เหลืออีก ÷ เหลือกี่วัน` · `คาดเสร็จ (ETA)` จากอัตราผลิตเฉลี่ยล่าสุด

แล้วจัดสถานะ (เลือกความรุนแรงสูงสุด) → เรียงงานด่วนขึ้นบน + แถบเตือนบนแดชบอร์ด:

| สถานะ | เงื่อนไข | สี |
|---|---|---|
| ✅ เสร็จแล้ว | ผลิตครบเป้า | เขียว |
| 🟢 ทันเวลา | คาดเสร็จ ≤ กำหนดส่ง | เขียว |
| 🟠 ใกล้ครบกำหนด | เหลือ ≤ 2 วัน ยังไม่ครบ | ส้ม |
| 🟠 คืบหน้าช้า | คืบหน้าจริงต่ำกว่าที่ควรเป็นตามเวลา > 15% | ส้ม |
| 🔴 เสี่ยงไม่ทัน | อัตราผลิตจริงทำไม่ทัน (ETA เลยกำหนด) | แดง |
| ⛔ เลยกำหนด | วันนี้ > กำหนดส่ง ยังไม่ครบ | แดง |

"งานด่วน" บนแถบเตือน = เลยกำหนด + เสี่ยงไม่ทัน + ใกล้ครบกำหนด

**PIN ผู้ดูแล** (ล้าง/นำเข้าทับ): `qcopr1234` — เปลี่ยนได้ที่ `CONFIG.ADMIN_PIN`

## ต่อออนไลน์ (Cloudflare Worker + D1)

```bash
# 1) สร้าง D1
wrangler d1 create qc-opr
#    เอา database_id ที่ได้ไปใส่ใน wrangler.toml

# 2) สร้างตาราง
wrangler d1 execute qc-opr --remote --file=./schema.sql

# 3) deploy Worker
wrangler deploy
#    จะได้ URL เช่น https://qc-opr.<subdomain>.workers.dev
```

จากนั้นในแอป → หน้า **ตั้งค่า** → ช่อง **API Base URL** ใส่ URL ของ Worker → กด *เชื่อมต่อ*
แอปจะดึงข้อมูลจากคลาวด์และซิงก์ทุกอุปกรณ์ที่ตั้ง URL เดียวกัน

> อยากเสิร์ฟ PWA จาก Worker เดียวกันด้วย: uncomment บล็อก `[assets]` ใน `wrangler.toml` แล้ว `wrangler deploy` ใหม่ — เปิด `https://qc-opr...workers.dev/` ได้เลย

### Endpoints
| Method | Path | ทำอะไร |
|---|---|---|
| GET | `/api/init` | คืน `{ machines, records, plans, valves, targets }` |
| POST | `/api/records` | upsert 1 record |
| DELETE | `/api/records/:id` | ลบ record |
| POST | `/api/plans` | upsert 1 แผนงาน |
| DELETE | `/api/plans/:id` | ลบแผนงาน |
| POST | `/api/valves` | upsert 1 valve (route/stages เป็น JSON) |
| DELETE | `/api/valves/:id` | ลบ valve |
| POST | `/api/targets` | upsert KV ตัวเลข (ช่องซิงก์ข้ามอุปกรณ์) |

## แพทเทิร์นซิงก์ข้ามอุปกรณ์ (machine → process)

process ของแต่ละเครื่องเป็น "ค่าตั้งค่าที่ต้องเห็นตรงกันทุกอุปกรณ์" จึงเก็บผ่าน KV store `targets`
ใต้ key `mcproc.<machineId>` โดย **เก็บเป็นตัวเลข** (ตาราง `targets.value` เป็น `INTEGER`):

- `procCode(name)` — แฮชชื่อ → เลขด้วย **FNV-1a** (mask 30 บิตกัน INTEGER ล้น)
- ถอดกลับด้วย `codeToName(code)` ที่ไล่แฮชทุกชื่อใน "คลังคำ" (`buildVocab()`) ซึ่งฝังในโค้ดทุกเครื่อง
- `pushMcProc()` = optimistic local + upsert ขึ้นคลาวด์ · `syncMcProcFromCloud()` = pull (cloud คือแหล่งความจริง) + migrate ค่า local-only ขึ้นครั้งแรก

เพิ่มชื่อ process ใหม่ที่ผู้ใช้พิมพ์เอง → เพิ่มเข้า "คลังคำ" ในหน้าตั้งค่า เพื่อให้เครื่องอื่นถอดรหัสได้
อ้างอิงเทคนิคเต็ม: `PATTERN-cross-device-kv-sync.md`
